## Context

当前并行热点集中在 `Rasterizer::RasterizeTriangles()`：
1) Tile bin 统计采用每线程局部直方图 + `critical` 合并，在线程数上升时出现串行瓶颈；
2) 若干循环使用 `dynamic,1`，在工作量较均匀时会产生较高调度开销；
3) 项目已在 CMake、编译选项和代码结构上深度绑定 OpenMP，短期替换并行框架会扩大回归面。

约束：
- 保持 C++20 + OpenMP，不新增 TBB 依赖；
- 保持双精度、现有渲染结果和公共 API 不变；
- 变更应聚焦性能关键路径并可回退。

## Goals / Non-Goals

**Goals:**
- 消除或显著降低 tile bin 阶段的串行归约开销。
- 在不改变渲染语义前提下，优化 OpenMP 调度策略，提升多核扩展性。
- 提供可控的并行参数与统计输出，支持按场景调优。
- 保证线程安全：不引入数据竞争，不破坏现有 tile 独占写入模型。

**Non-Goals:**
- 不切换到 TBB/oneTBB。
- 不重写渲染管线架构（Pass 系统、材质系统等）。
- 不改变 `Renderer` 对外接口与场景数据流。

## Decisions

### Decision 1：保留 OpenMP，仅重构归约与调度
- 方案 A（采用）：继续 OpenMP，重构热点循环。
- 方案 B（不采用）：迁移到 TBB。
- 选型理由：
  - A 的改动面小、验证成本低，能直接针对当前瓶颈（`critical`）下刀；
  - B 需要重写并行模式与构建依赖，短期风险高于预期收益。

### Decision 2：将 bin 计数归约从 `critical` 改为“分块归约”
- 方案 A（采用）：每线程维护局部计数，按 tile 块并行归约（避免全局 `critical`）。
- 方案 B（备选）：每次计数使用原子加（`atomic`）。
- 选型理由：
  - A 在大 tile 数量下更稳定，减少锁竞争与缓存抖动；
  - B 原子冲突在热点 tile 上可能明显，吞吐不稳定。
- 量级估算：
  - 现状 `critical` 合并在 8~16 线程时可能产生 $0.3\sim1.5\,ms$ 串行段；
  - 分块归约目标将该段压到 $<0.2\,ms$（1080p 常见场景）。

### Decision 3：按阶段选择调度策略并允许配置
- 几何构建/Tile 光栅：保留 `dynamic` 或 `guided`（负载不均场景）。
- 清屏/后处理/规则循环：使用 `static`（最小调度开销）。
- 提供配置入口（编译时默认 + 运行时可选覆盖），用于 chunk 调优。
- 量级估算：
  - `dynamic,1` 单任务调度成本通常在 $\mu s$ 量级，任务过细时可放大为 $0.1\sim0.4\,ms$ 帧成本；
  - 增大 chunk 或改 `guided` 预期可回收其中 30%~60%。

### Decision 4：线程安全保持“tile 独占写入 + 原子统计”
- 颜色/深度写入仍由单线程独占 tile，避免像素级锁。
- 全局统计继续使用 `atomic` 或线程局部汇总后合并。
- 明确禁止在热点路径引入 `std::mutex`。

## Risks / Trade-offs

- [不同 CPU 架构最佳调度参数差异大] → 通过配置化和基准场景选默认值，并保留回退开关。
- [分块归约实现复杂度上升] → 先做最小可行实现（仅替换 `critical`），配合单元/回归对比。
- [过度调参导致可维护性下降] → 仅暴露少量参数（policy/chunk），其余保持内部固定。
- [统计开销影响结果] → 统计默认低频输出，详细 profiling 仅在调试开关开启时启用。

## Migration Plan

1. 在 `Rasterizer` 中替换 bin 计数归约路径（去 `critical`）。
2. 统一并行循环调度策略的常量与配置入口。
3. 增加阶段级计时与并行统计输出（构建/bin/raster）。
4. 对比基线（当前主分支）进行功能回归与性能评估。
5. 若某场景退化超过阈值（如 >5%），通过配置快速回退到旧调度策略。

回滚策略：
- 保留旧实现路径的条件编译或开关，必要时一键切回旧策略。
